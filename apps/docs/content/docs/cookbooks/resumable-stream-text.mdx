---
title: Resumable streamText (event log)
description: Cache model stream events under a resumeKey, replay slices, and optionally continue upstream with streamText.
---

`streamText` can persist each `ModelStreamPart` as it flows, then replay from
a stored index and optionally continue live generation from upstream when the
session is not yet completed. Use `continueUpstream: true` with the same model
inputs (`model` + prompt/messages + options) to opt in.

## Replay then continue upstream

If a client disconnects mid-stream, replay from the first missing event index
and continue generation in a single stream:

```ts
const resumed = await streamText({
  resumeKey: out.resumeKey!,
  resumeFromIndex: clientLastSeenIndex + 1,
  streamEventStore: store,
  continueUpstream: true,
  model,
  prompt: 'Hi',
});
```

When the stored log is already complete (`finish`/`error` recorded),
`continueUpstream: true` becomes a no-op for upstream and only replays cached
parts.

## In-memory (tests, single process)

```ts
import {
  InMemoryResumableStreamEventStore,
  streamText,
} from '@ziro-agent/core';

const store = new InMemoryResumableStreamEventStore();

const out = await streamText({
  model,
  prompt: 'Hi',
  resumable: true,
  streamEventStore: store,
});
// out.resumeKey — opaque, treat like a session handle
const text = await out.text();

// Replay from event index 0 (full run)
const again = await streamText({
  resumeKey: out.resumeKey!,
  resumeFromIndex: 0,
  streamEventStore: store,
});
await again.text();
```

`resumeFromIndex` is **0-based**: use the first index the client has **not** yet
applied (after a disconnect, pass “last seen index + 1” if you only need the tail).

## Redis (TTL, multi-process readers)

Use `RedisResumableStreamEventStore` from `@ziro-agent/checkpoint-redis` with the
same `ResumableStreamEventStore` interface as the in-memory store. Default key
prefix is `ziro:st` (separate from the checkpointer’s `ziro:cp`).

```ts
import { streamText } from '@ziro-agent/core';
import IORedis from 'ioredis';
import { fromIoRedis, RedisResumableStreamEventStore } from '@ziro-agent/checkpoint-redis';

const redis = new IORedis(process.env.REDIS_URL!);
const streamEventStore = new RedisResumableStreamEventStore({
  client: fromIoRedis(redis),
  ttlSeconds: 3600,
  maxEventsPerStream: 10_000,
  maxBytesPerStream: 8 * 1024 * 1024, // 8 MiB
});

const out = await streamText({
  model,
  prompt: 'Hi',
  resumable: true,
  streamEventStore,
});
```

## Memory guard (in-memory store)

Use caps to prevent unbounded growth in long streams:

```ts
import { InMemoryResumableStreamEventStore } from '@ziro-agent/core';

const streamEventStore = new InMemoryResumableStreamEventStore({
  maxEventsPerStream: 10_000,
  maxBytesPerStream: 8 * 1024 * 1024, // 8 MiB
});
```

When a cap is exceeded, `streamText(...).text()` / iterators reject with
`ResumableStreamError` so you can log and retry without resumable mode.

Redis-backed stores support the same per-stream caps (`maxEventsPerStream`,
`maxBytesPerStream`) and an optional `measurePartBytes` hook (default: UTF-8
length of the stored JSON line). You should still use TTL (`ttlSeconds`) as the
primary guardrail for long-lived sessions.
